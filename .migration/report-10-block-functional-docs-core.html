1. Hero

7 VariationsComplexity: Simple–MediumEst: 3–5 dev daysPriority: P0
Block Overview: The Hero block is the primary page-heading component appearing at the top of virtually every page. It sets the visual tone with background imagery, text overlays, and optional CTAs. Variations span all three AEM platforms (Modern, Legacy, Corporate) with significant DOM differences between each. EDS consolidation target: 2–3 hero variants covering all use cases.

V1: Page Heading (Standard)

Modern

Full-width background image with white text overlay containing page title (h1) and optional subtitle. The most common hero pattern across Modern AEM pages, used on category landing pages and informational pages.

Structure / Fields

ElementTag/ClassDescription
Container.cmp-p-pageheadingheroOuter wrapper for the hero component
Background Image.cmp-p-pageheadinghero__imageFull-width background image, CSS background-image or <picture>
Content Overlay.cmp-p-pageheadinghero__contentText container positioned over the image
Titleh1Page title, white text, typically 32–48px
SubtitlepOptional descriptive subtitle below h1
Breadcrumb.cmp-breadcrumbOptional breadcrumb trail above or below hero

Visual / Functional Differences

Simplest hero variant. No CTA buttons, no foreground images, no gradient overlay. Distinguished from V3/V4 by lack of gradient and buttons. Height ranges 300–400px. Text is left-aligned with max-width constraint.

Acceptance Criteria

  • Hero renders full-width with background image covering the entire container
  • H1 title displays in white text with adequate contrast ratio (≥4.5:1) against background
  • Background image is responsive and uses srcset or picture element for optimal loading
  • Hero height is consistent (300–400px) across standard viewport widths
  • Subtitle text renders below title when provided, hidden when empty
  • Breadcrumb navigation renders correctly above or below the hero if present
  • No horizontal overflow or content clipping on any viewport width

Interactions

Click
No interactive click targets within V1 hero (no CTAs/links)
Hover
No hover states. Cursor remains default.
Animation
Optional: subtle fade-in on page load via CSS opacity transition (0→1 over 0.3s)
Lazy Loading
Hero image should use eager loading (loading="eager") as it's above the fold (LCP candidate)

Responsive Behavior

Mobile
< 768px
  • Height reduces to 200–250px
  • Title font-size: 24px
  • Text centered
  • Background image repositioned (object-position: center)
Tablet
768–1024px
  • Height: 280–350px
  • Title font-size: 32px
  • Text left-aligned
  • Full image visible
Desktop
> 1024px
  • Height: 300–400px
  • Title font-size: 40–48px
  • Text left-aligned with max-width ~60%
  • Full bleed background

Accessibility

  • Background image must have meaningful alt text or be marked decorative (role="img" aria-label="...")
  • H1 is the single page heading — only one h1 per page
  • Color contrast: white text on image must meet WCAG 2.1 AA (4.5:1 minimum)
  • Text remains readable if images fail to load (dark fallback background-color)
  • No keyboard traps — hero is non-interactive content

Authoring Requirements

Background ImageRequired *Image (DAM asset)Min 1600×400px, max 500KB optimized
Title (H1)Required *TextMax 80 characters, no HTML
SubtitleOptionalRich TextMax 200 characters
Mobile ImageOptionalImage (DAM asset)Min 768×300px, portrait crop recommended

V2: Foreground Product Image

Modern

Background image with a foreground product image floated to the right and text overlay on the left. Includes CTA buttons below the text. Used for product/technology landing pages where a hero product shot is needed.

Titleh1Product/technology name
DescriptionpProduct tagline or brief description
CTA Buttons.cmp-p-pageheadinghero__cta a.btn1–2 action buttons (primary + secondary)

Visual / Functional Differences

Distinguished from V1 by the foreground product image and CTA buttons. The hero area is split ~60/40 (text/image). Product image has transparent background (PNG) and may extend beyond the hero container bottom edge for a "pop-out" effect.

Acceptance Criteria

  • Product image renders on the right side with transparent background preserved
  • Text content and product image do not overlap on desktop viewports
  • CTA buttons are visible, clickable, and have proper hover states
  • Product image scales proportionally without distortion
  • On mobile, layout stacks vertically: text on top, product image below
  • Both background and product images load with appropriate priority

Interactions

Click
CTA buttons navigate to target URLs. Primary button has solid fill, secondary has outline style.
Hover
CTA buttons: background color darkens, subtle box-shadow appears. Product image: no hover effect.
Animation
Product image may slide in from right on page load (translateX animation). Text fades in.
Dynamic
Product image may be served via Scene7/Dynamic Media for on-the-fly transformations.

Responsive Behavior

Mobile
< 768px
  • Stacked layout: text block on top
  • Product image below, centered, max-width 80%
  • CTA buttons full-width, stacked
Tablet
768–1024px
  • Side-by-side maintained but narrower gap
  • Product image ~35% width
  • Smaller text sizes
Desktop
> 1024px
  • 60/40 split text/image
  • Product image may overflow bottom
  • Full typography scale

Accessibility

  • Product image must have descriptive alt text (product name + context)
  • CTA buttons must have clear, descriptive link text (not "Click here")
  • Keyboard focus order: h1 → description → CTA primary → CTA secondary
  • Focus visible outlines on all interactive elements

Acceptance Criteria

  • Primary CTA button is visually prominent with solid fill
  • Secondary CTA has clear outline style differentiating it from primary
  • Both buttons are clickable and navigate to correct URLs
  • Buttons maintain minimum touch target size (44×44px)
  • Button text is readable against button background

Click
Primary CTA: navigates to product/signup page. Secondary CTA: opens resource or alternative path.

2. Cards

11 VariationsComplexity: Simple–MediumEst: 5–8 dev daysPriority: P0
Block Overview: Cards are the most versatile and frequently used content block across the site. They present content in grid layouts with combinations of images, titles, descriptions, and CTAs. Found on virtually every page type. 11 variations span Modern and Legacy platforms with significant layout differences. EDS consolidation target: 4–5 card variants using CSS modifiers.

V1: Image + Title + Desc + CTA (3-up)

Modern

Three cards per row, each with top image, h3 title, paragraph description, and CTA link. The standard card layout for most Modern AEM pages. ~33% width per card.

Structure / Fields

ElementTag/ClassDescription
Grid Container.cmp-p-cardsFlex/grid wrapper, 3-column
Card.cmp-p-cards__itemIndividual card container
Image.cmp-p-cards__image imgTop image, 16:9 or 4:3 ratio
Titleh3Card heading, linked
DescriptionpBrief text, 2–3 lines
CTAa.cmp-p-cards__cta"Learn more" style link

Acceptance Criteria

  • 3 cards render in a single row on desktop with equal widths
  • Card images maintain consistent aspect ratio across all cards in a row
  • Cards have equal height (CSS flexbox/grid stretch)
  • CTA link is positioned at the bottom of each card consistently
  • Entire card is clickable (card-as-link pattern) or CTA link only
  • Cards wrap to next row if more than 3 items
  • Empty/missing image gracefully shows placeholder or collapses

Interactions

Click
Card click navigates to detail page. CTA link is primary click target.
Hover
Card: subtle box-shadow elevation (0 4px 12px rgba(0,0,0,0.15)). CTA: underline + color shift.
Lazy Load
Card images use loading="lazy" if below fold.

Responsive Behavior

Mobile
< 768px
  • 1 card per row, full-width
  • Stacked vertically
  • 16px gap between cards
Tablet
768–1024px
  • 2 cards per row
  • Last card wraps to new row if 3 items
Desktop
> 1024px
  • 3 cards per row
  • 24px gap
  • Max-width 1200px container

Accessibility

  • Card images have descriptive alt text
  • H3 headings maintain proper heading hierarchy
  • If card-as-link, use aria-label on card container with full card title
  • Focus visible on CTA links and any clickable areas
  • Card grid uses semantic list markup (<ul>/<li>) or role="list"
